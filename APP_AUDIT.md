# VitalIQ Application: Functional Audit Report
**Date:** May 1, 2026
**Status:** Post-Migration & AI Upgrade Phase

## 1. Identified Functional Issues

### 🔴 High Priority: Connectivity & Environment
*   **Hardcoded API URLs:** The frontend `api.js` relies on a hardcoded local IP address (e.g., `192.168.31.167`). If the laptop changes WiFi or the router reassigns an IP, the mobile app will immediately lose connection ("Network Error").
*   **Environment Fragmentation:** There are multiple `.env` files and cross-directory loading (AI service loading from Backend folder). This creates a "single point of failure" if the folder structure is moved.

### 🟡 Medium Priority: Database & State
*   **Migration Fragmentation:** The database schema was fixed using 5+ separate migration scripts (`migrate_streak.js`, `migrate_onboard.js`, etc.). A fresh installation using only `init.sql` will fail because `init.sql` is now outdated compared to the actual app logic.
*   **Onboarding Logic:** The `users` table depends on several columns (BMR, TDEE, City, State) being calculated correctly. If the frontend sends invalid data for any of these, the onboarding flow may crash without a clear error message to the user.

### 🟡 Medium Priority: AI Service Reliability
*   **Model Volatility:** As seen today, external AI providers (Groq/Gemini) decommission models frequently. The application currently has no "Auto-Fallback" to a local model if the Cloud API fails.
*   **Timeout sensitivity:** Large images (high-res photos from modern phones) take longer to upload and process. The current 30s timeout is better but might still fail on slow rural 4G/5G connections.

### 🟢 Low Priority: User Experience
*   **Data Mismatch:** The AI (Llama 4) is very smart and might return "Paneer Tikka Masala," but the local `foods.json` might only have "Paneer." The current logic "fuzzy matches" them, but this can lead to calorie inaccuracies.
*   **Auth Persistence:** If the JWT secret is changed in `.env`, all currently logged-in mobile users will be kicked out without an "Expired Session" warning.

## 2. Recommended Next Steps (Non-Coding)

1.  **Unified Schema:** Consolidate all `migrate_*.js` changes back into a single `schema_v2.sql` for future deployments.
2.  **Service Discovery:** Consider a way to dynamically detect the server IP or use a tunnel (like Ngrok) for stable development.
3.  **Local Fallback:** Re-introduce a lightweight local model (like the MobileNet we had) as a "backup" for when the internet is slow or the API is down.
4.  **Error Boundaries:** Ensure the mobile app shows specific errors (e.g., "AI Busy," "Invalid Food," "No Internet") instead of a generic "Analysis Failed."
